## 2. 已确认进入 MVP（v0.1）的需求

（节选自小黄条需求调研 + 原型已确认）

1. 透明/半透明嵌入式桌面窗口，**始终置顶**
2. 拖动**顶部栏**移动窗口；**四边 + 四角**悬停拖动缩放（标准窗口手感）
3. **点击列表空白处**新增待办（无固定底部输入框）

5. 每条待办有「**详情**」按钮（🔧）：标题、类型（普通 / 日期倒数）、目标日期、标签、备注
6. **标签系统**（带颜色）+ **多清单**切换
7. 设置面板：窗口透明度、开机自启
8. 数据存**本地 JSON 文件**；技术栈 **Tauri + 原生 TypeScript + MIT**（体积小、免代码签名、易修改）

## 3. 需求池（讨论中 / 待规划的 New Ideas）

### REQ-001　一级内容标签 / 二级内容标签（层级结构）

- **提出人**：M2　**提出日期**：2026-07-14　**状态**：`已确认待排期`（目标 v0.2）
- **原始想法**：希望可以有一级内容标签和二级内容标签。例如一级内容是「完成家务事项」，二级内容是在一级下面有「洗碗、扫地、拖地、洗衣服」等事项。
- **初步理解（待 M2 确认）**：本质是一种**层级 / 分组结构**，有两种可能的实现形态：
  - **形态 A（任务层级）**：一级是一个「父待办 / 分组」，二级是它的「子任务」。父项可整体完成、可展开 / 收起。
  - **形态 B（标签层级）**：一级 / 二级是两级标签，给扁平待办做分类（如 标签＝家务 → 子标签＝洗碗）。

#### 两种形态对比

| 维度 | 形态 A：任务层级（推荐 ✅） | 形态 B：标签层级 |
|------|--------------------------|------------------|
| 一级 | 「父待办 / 分组」，可整体完成、可展开 / 收起 | 一级标签（大分类），如「家务」 |
| 二级 | 「子任务」，从属于父项的具体执行项 | 二级标签（子分类），如「洗碗」 |
| 待办结构 | 有层级嵌套，父子从属 | 仍扁平，仅用两级标签分类筛选 |
| 口语契合度 | 高：「完成家务」自然对应父子关系 | 中：偏「分类」语义 |
| 与现有功能区分 | 清晰（多清单＝文档，标签＝归类，层级＝拆解） | 较低（与现有标签概念重叠） |

#### 开放问题 + 已确认结论（2026-07-14 M2 拍板）

| # | 问题 | 确认结论 | 状态 |
|---|------|----------|------|
| Q1 | 一级和二级是「分组容器」还是「标签」？ | **分组容器**（层级 / 分组结构，父待办 → 子任务） | ✅ 已确认 |
| Q2 | 二级事项是否独立勾选完成？父项完成是否自动完成所有子项？ | 二级**可独立勾选**；父项完成**自动完成**所有子项 | ✅ 已确认 |
| Q3 | 是否需要无限层级，还是固定两级就够？ | 支持**最多五个层级**（不限定两级） | ✅ 已确认 |
| Q4 | 与现有「多清单」「标签」功能如何区分，避免功能重叠？ | 见下方概念边界 | ✅ 已确认 |

#### 概念边界（避免功能重叠）

```
多清单（Multiple Lists）
└─ 类比：新开一个文档 / 笔记本（如 工作清单、生活清单、运动清单）
└─ 特点：完全独立的待办集合

标签（Tags）
└─ 类比：文档内的分类标记（如 #紧急 #居家）
└─ 特点：同一清单内任务可打多个标签做归类

层级 / 任务拆解（Hierarchy）← 本需求
└─ 类比：大纲 / WBS 工作分解
└─ 特点：父子从属关系，联动完成状态
```

- **最终结论（M2 已拍板）**：采用**形态 A（任务层级）**，贴合"完成家务事项"的口语表达；上述 Q1–Q4 结论全部确认生效。
