## 2. 已确认进入 MVP（v0.1）的需求

（节选自小黄条需求调研 + 原型已确认）

1. 透明/半透明嵌入式桌面窗口，**始终置顶**
2. 拖动**顶部栏**移动窗口；**四边 + 四角**悬停拖动缩放（标准窗口手感）
3. **点击列表空白处**新增待办（无固定底部输入框）
4. 每条待办可：勾选完成、置顶、删除
5. 每条待办有「**详情**」按钮（🔧）：标题、类型（普通 / 日期倒数）、目标日期、标签、备注
6. **标签系统**（带颜色）+ **多清单**切换
7. 设置面板：窗口透明度、开机自启
8. 数据存**本地 JSON 文件**；技术栈 **Tauri + 原生 TypeScript + MIT**（体积小、免代码签名、易修改）

## 3. 需求池（讨论中 / 待规划的 New Ideas）

### REQ-001　一级内容标签 / 二级内容标签（层级结构）

- **提出人**：M2　**提出日期**：2026-07-14　**状态**：`讨论中`（未排期，目标 v0.2）
- **原始想法**：希望可以有一级内容标签和二级内容标签。例如一级内容是「完成家务事项」，二级内容是在一级下面有「洗碗、扫地、拖地、洗衣服」等事项。
- **初步理解（待 M2 确认）**：本质是一种**层级 / 分组结构**，有两种可能的实现形态：
  - **形态 A（任务层级）**：一级是一个「父待办 / 分组」，二级是它的「子任务」。父项可整体完成、可展开 / 收起。
  - **形态 B（标签层级）**：一级 / 二级是两级标签，给扁平待办做分类（如 标签＝家务 → 子标签＝洗碗）。
- **开放问题（需与 M2 讨论）**：
  1. 一级和二级是「分组容器」还是「标签」？（决定交互模型）
  2. 二级事项是否独立勾选完成？父项完成是否自动完成所有子项？
  3. 是否需要无限层级，还是固定两级就够？
  4. 与现有「多清单」「标签」功能如何区分，避免概念重叠？
- **我的初步建议**：倾向先按**形态 A（任务层级）**做，更贴合"完成家务事项"这种口语表达；但仍等 M2 拍板。
